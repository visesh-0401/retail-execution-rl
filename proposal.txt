# Proposal: Constrained Reinforcement Learning for Retail Order Execution with Broker API Limitations

## 1. Title

**Optimizing Order Splitting Under API Rate Limits: A Reinforcement Learning Approach for Retail Traders**

---

# 2. Problem Statement

### The Retail Execution Gap

Institutional algorithmic trading has sophisticated execution engines, but **retail traders face fundamentally different constraints**:

* **API rate limits** (typically 1-10 requests/second from retail brokers)
* **Discrete order sizing** (cannot split into 100s of micro-orders)
* **Latency constraints** (no direct exchange connectivity, 100ms+ delays)
* **Order rejection risk** (brokers reject suspicious patterns)

Existing literature (Nevmyvaka 2006, Hafsi 2024, Kim 2023) assumes **institutional infrastructure** with:
- Direct exchange access
- Microsecond latency
- Unlimited order submission frequency
- No interaction penalties

**The gap**: There is **no published research** on RL-based execution optimized for retail broker API constraints.

The research problem:

> Can reinforcement learning learn optimal order splitting policies **within broker API rate limits** while minimizing execution slippage—a problem fundamentally different from institutional execution?

---

# 3. Objectives

The project aims to develop and validate an **RL-based execution optimizer for retail traders** that:

1. Operates within realistic retail broker **API rate limit constraints** (1-10 requests/second).
2. Learns optimal order splitting policies across **discrete order sizes** (whole shares/contracts).
3. Predicts **execution slippage** given API constraints as a feature.
4. Outperforms baseline strategies (TWAP, VWAP, random splits) in simulated retail broker environment.
5. Publishes findings in a **quantitative finance / ML conference** (NeurIPS, ICML workshop, or q-fin journal).

---

# 4. Key Research Questions

1. **Does API rate limiting fundamentally change optimal execution strategy?** Can an RL agent learn policies that differ from institutional assumptions?

2. **How much slippage is attributable to API rate limits vs. market microstructure?** Decompose execution cost into two components.

3. **Can PPO (policy gradient) outperform TWAP/VWAP baselines under discrete order constraints?** What is the performance gap?

4. **Does the learned policy generalize across different brokers, assets, and market conditions?** (stress test generalization)

5. **What is the minimum API request frequency needed to match institutional execution quality?** Find the threshold.

# 6. System Modules

## 6.1 Retail-Constrained Execution Simulator

**Purpose**: Model realistic retail broker execution constraints.

**Input**: Historical OHLCV data + order book snapshots (from public sources like Alpaca, EODHD, or Yahoo Finance).

**Features modeled**:

| Constraint              | Modeling                               |
| ----------------------- | -------------------------------------- |
| API Rate Limit          | Max N requests per second (configurable) |
| Discrete Order Size     | Rounds to whole shares/contracts        |
| Execution Latency       | Random 50-500ms delay                  |
| Market Impact           | Simple linear model (qty × volume %)   |
| Bid-Ask Spread          | Historical average from data           |
| Order Rejection         | Random 2-5% rejection rate on resubmits |

**Core simulation loop**:
```
while remaining_quantity > 0:
    if API_rate_limit_allows():
        submit_order(next_split_size)
        remaining_qty -= execution_qty
        slippage += (execution_price - mid_price)
    else:
        wait_for_API_slot()
```

---

## 6.2 Reinforcement Learning Agent (PPO)

**Algorithm**: Proximal Policy Optimization (PPO)

**Why PPO?** Simple, sample-efficient, proven in finance (Kim et al. 2023), implementable in 3 months.

### State Representation

```python
state = {
    'remaining_quantity': float,          # % of original order left
    'elapsed_time': float,                # % of execution window used
    'bid_ask_spread': float,              # current spread (bps)
    'volatility_5min': float,             # rolling 5min volatility
    'api_requests_used': int,             # how many API calls made
    'api_requests_limit': int,            # rate limit per second
    'market_momentum': float,              # price momentum indicator
    'time_until_deadline': float          # time remaining
}
```

### Action Space

**Discrete actions**: Order split sizes relative to remaining qty

```
Actions: [submit 10%, 20%, 30%, 50%, 75%, 100% of remaining]
```

(Simple discrete space → easier to train in 3 months)

### Reward Function

```
reward = -slippage - transaction_cost - api_rate_penalty

where:
  slippage = (execution_price - mid_price) * qty
  transaction_cost = bid_ask_spread * qty
  api_rate_penalty = 10 * (requests_used - budget)
                     if over budget else 0
```

**Rationale**: Penalizes both market costs AND excessive API usage.

### Training Configuration

- Environment: Gym-compatible custom environment
- Training episodes: 5,000 episodes minimum
- Batch size: 64
- Learning rate: 3e-4
- Training duration: ~4 hours on GPU

## 6.3 Backtesting & Evaluation

**Baselines to compare against**:

| Baseline         | Description                  | Metric                    |
| ---------------- | ---------------------------- | ------------------------- |
| Market Order     | Single execution               | 100% slippage (reference) |
| TWAP (w/ limits) | Equal splits, respecting API limit | Standard benchmark     |
| VWAP (w/ limits) | Volume-weighted splits, respecting API limit | Volume-aware benchmark    |
| Random Splits    | Uniformly random order splits  | Lower bound performance   |

**Evaluation metrics**:

```
1. Total Execution Cost (bps) = slippage + transaction costs
2. Cost Efficiency = (RL_cost - VWAP_cost) / VWAP_cost (% improvement)
3. API Efficiency = Total cost / API requests used
4. Robustness = Performance across 10 different stocks/assets
5. Generalization = Training on stock A, test on stock B (out-of-sample)
```

**Test scenarios**:

- **Normal conditions**: Typical market volatility
- **High volatility**: Days with +2σ volatility spike
- **Low liquidity**: Penny stocks or off-hours trading
- **API degradation**: Simulate API rate limits of 1, 5, 10 rps
